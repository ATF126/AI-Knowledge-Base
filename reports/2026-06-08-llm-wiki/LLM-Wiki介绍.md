# Karpathy 的 LLM-Wiki：让知识库成为可复利资产

## 一句话概括

LLM-Wiki 是 Andrej Karpathy 提出的一个知识管理模式：把原始资料交给 LLM 处理，让 LLM 持续维护一个结构化、可链接、可追溯的 Markdown Wiki。它不是简单的“问文档”或“RAG 搜索”，而是让知识在每次导入、查询和整理后沉淀下来，形成长期可复用的知识层。

原文链接：[karpathy/LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 它想解决什么问题

传统个人知识库和团队 Wiki 经常失败，不是因为大家不知道要记录，而是因为维护成本太高：

- 新资料来了，要摘要、归类、打标签、补链接。
- 旧页面过时了，要更新、合并、拆分。
- 同一个概念散落在多个地方，要保持一致。
- 查询时需要跨多个来源综合判断。

这些工作对人来说琐碎、重复、容易拖延，但正好是 LLM 擅长的工作。LLM-Wiki 的核心判断是：让人负责选择资料、提出问题、判断方向；让 LLM 负责整理、链接、同步、查漏和维护。

## 和传统 RAG 的区别

传统 RAG 通常是在提问时临时检索资料，然后把相关片段塞给模型生成答案。它的优势是实现直接，缺点是每次都像重新做一次研究：系统不一定记住上次的综合结果，也不一定会主动更新知识结构。

LLM-Wiki 的思路不同。它在原始资料和问答之间增加一层“人工可读、机器可维护”的 Wiki：

- 原始资料不变，作为事实来源。
- LLM 把资料消化成结构化页面。
- 页面之间通过链接组织起来。
- 后续新资料会更新已有页面，而不是只生成一次性答案。
- 有价值的查询结果也可以写回 Wiki，变成新的知识资产。

因此，LLM-Wiki 更像“知识编译器 + 长期记忆系统”，而不是一次性的检索增强问答。

## 三层架构

LLM-Wiki 通常由三层组成。

### 1. Raw Sources：原始资料层

原始资料层保存不可变的事实来源，例如：

- 论文、文章、网页摘录
- 书籍笔记
- 会议记录
- 图片、音频、视频转录
- 数据文件

这层的原则是“只读不改”。它是 Wiki 的证据基础，也是后续追溯来源的依据。

### 2. Wiki Pages：知识层

Wiki 是 LLM 维护的结构化知识网络。常见页面类型包括：

- Source Summary：单个来源的摘要页
- Entity Page：人物、项目、组织、产品等实体页
- Concept Page：概念、方法、理论、模型页
- Comparison Page：对比分析页
- Overview Page：主题综述和综合页

这一层不是原始资料的复制，而是经过整理、链接和综合后的知识层。

### 3. Schema：规则层

Schema 是给 LLM 的操作说明，通常写在 `CLAUDE.md`、`AGENTS.md`、`TheSchema.md` 这类文件里。它定义：

- 目录结构
- 页面命名规则
- frontmatter 格式
- 引用规范
- Ingest、Query、Lint 的工作流
- 什么时候可以自动修改，什么时候需要先确认

Schema 的作用是把 LLM 从“聊天助手”变成“有纪律的 Wiki 维护者”。

## 三个核心操作

### 1. Ingest：导入资料

当加入一份新资料时，LLM 不只是做摘要，而是要把它编入现有知识库：

1. 阅读原始资料。
2. 创建或更新来源摘要页。
3. 识别相关人物、项目、概念和事件。
4. 更新已有实体页、概念页或综述页。
5. 添加内部链接和来源引用。
6. 更新索引和日志。

一次好的 Ingest 可能会影响多个页面，因为新资料往往不只是一个孤立事实，而是会改变已有知识网络。

### 2. Query：基于 Wiki 回答问题

查询时，LLM 应该先阅读索引和相关页面，再给出综合回答。与普通聊天不同，它的回答应基于 Wiki 中已有的结构化知识，并尽量带上来源或页面引用。

如果某次回答很有价值，例如形成了一个新的对比、判断框架或主题综述，就可以把它写回 Wiki，成为新的页面。

### 3. Lint：健康检查

Lint 是对 Wiki 的定期维护。LLM 可以检查：

- 哪些页面互相矛盾
- 哪些结论被新资料推翻
- 哪些页面没有入链或出链
- 哪些概念反复出现但没有独立页面
- 哪些页面需要合并、拆分或补引用

这让知识库不会随着时间变成一堆旧笔记，而是持续保持可用。

## 两个关键文件

### `index.md`

`index.md` 是内容索引，帮助人和 LLM 快速理解 Wiki 中有什么。它通常按页面类型、主题或标签组织，并为每个页面提供一句话摘要。

### `log.md`

`log.md` 是操作日志，记录每次导入、查询、整理和检查。它让知识库的演化过程可追踪，也方便排查某个页面为什么被创建或修改。

## 适用场景

LLM-Wiki 适合那些需要长期积累、反复查询、不断综合的知识场景，例如：

- 个人知识管理
- 读书和论文研究
- 产品调研和竞品分析
- 技术选型和架构决策
- 投资、尽调和行业研究
- 课程学习和学习笔记
- 团队内部知识库
- 个人目标、健康、项目和生活管理

它尤其适合“资料越来越多，但希望理解越来越清晰”的场景。

## 实践建议

1. 保持原始资料不可变。不要让 LLM 直接改 `raw/` 中的内容。
2. 从少量固定页面类型开始，不要一开始设计过度复杂的分类。
3. 明确写好 Schema，让 LLM 知道如何命名、引用、链接和记录。
4. 每次导入资料后，要求 LLM 更新相关旧页面，而不是只创建新页面。
5. 定期执行 Lint，把知识库当成一个需要维护的系统。
6. 把高质量问答写回 Wiki，让查询本身也成为知识积累的一部分。

## 总结

Karpathy 的 LLM-Wiki 不是在讨论某个具体软件，而是在提出一种新的知识管理范式：让 LLM 维护一个可读、可编辑、可追溯、会持续演化的知识层。

它的关键价值在于把知识管理从“一次性问答”变成“长期积累”。每次导入资料、提出问题、整理矛盾，都会让 Wiki 更完整、更清晰、更有价值。换句话说，LLM-Wiki 的目标不是让你少写几条笔记，而是让知识库真正变成一个会复利增长的系统。
