Andrej Karpathy Skills 项目解读报告

一、项目背景

今天介绍的项目是 GitHub 上的 andrej-karpathy-skills。这个项目的定位很明确：它不是一个传统意义上的代码库，而是一个用于约束 AI 编程助手行为的规则文件集合。

项目描述中说明，它的核心是一个 CLAUDE.md 文件，用于改善 Claude Code 的代码生成行为，其思想来源于 Andrej Karpathy 对大语言模型编程常见问题的观察。

简单来说，这个项目想解决的问题是：
现在的 AI 编程助手虽然能写代码，但经常会过度假设、过度设计、修改无关代码，或者在没有验证的情况下直接给出结果。

因此，这个项目通过一套明确的编码行为规范，让 AI 在执行代码任务时更加谨慎、简洁和可控。


二、项目要解决的问题

在实际使用 AI 写代码时，经常会遇到几个问题。

第一，AI 容易在需求不明确时自行假设。
例如用户只是说“修复这个 bug”，AI 可能没有先确认 bug 的边界，就直接修改大量代码。

第二，AI 容易过度设计。
一个简单函数本来几十行就能完成，但 AI 可能会引入复杂类、配置文件、抽象接口，导致代码变得不必要地复杂。

第三，AI 容易修改无关代码。
比如只需要改一个函数，它却顺手重构其他模块、调整格式、删除注释，使得代码 diff 变得很大，后续很难审查。

第四，AI 经常缺少验证过程。
它可能只给出一个看起来合理的修改，但没有测试、没有复现问题，也没有明确说明这个修改是否真的解决了问题。

所以，这个项目的核心价值就是：
把 AI 编程从“直接生成代码”转变为“有约束、有目标、有验证的代码修改过程”。


三、项目核心内容

这个项目的核心文件是 CLAUDE.md。该项目直接提供了 CLAUDE.md 文件，用于作为 Claude Code 的行为指导文件。

其中最重要的是四条原则：


1. Think Before Coding：先思考，再写代码

第一条原则是 Think Before Coding，也就是写代码之前先理解任务。

它要求 AI 不要一上来就改代码，而是先明确：

- 用户真正想解决什么问题；
- 当前代码中哪些文件和模块相关；
- 是否存在需求歧义；
- 需要做哪些最小修改；
- 修改完成后应该如何验证。

这条规则的意义是避免 AI 在没有理解上下文的情况下盲目生成代码。

例如，用户说“优化这个模块”，AI 应该先区分：
这是性能优化、结构优化、可读性优化，还是 bug 修复。
如果不区分清楚，就很容易改偏。


2. Simplicity First：优先简单方案

第二条原则是 Simplicity First，也就是优先采用最简单的实现。

它要求 AI：

- 不要添加用户没有要求的功能；
- 不要引入不必要的依赖；
- 不要为了未来可能的需求提前设计复杂架构；
- 不要把简单问题写成复杂框架。

这条规则主要是限制 AI 的“过度工程化”倾向。

在科研代码或实验脚本中，这一点尤其重要。
因为很多代码的目标不是构建大型工程，而是稳定复现实验。如果 AI 随便引入抽象和依赖，反而会破坏可复现性。


3. Surgical Changes：最小范围修改

第三条原则是 Surgical Changes，可以理解为“外科手术式修改”。

也就是说，AI 只应该修改完成任务所必须修改的地方。

它要求：

- 不重构无关代码；
- 不调整无关文件格式；
- 不顺手删除看起来没用的代码；
- 尽量保持原有代码风格；
- 只清理本次修改引入的无用变量、函数或 import。

这条原则非常适合真实项目开发，因为真实项目最怕 AI “顺手大改”。
一旦 diff 很大，就很难判断哪些修改是必要的，哪些修改可能引入了新 bug。

对于我们做科研项目也一样。比如修改评估脚本时，只应该改指标计算相关代码，不能顺手改变数据读取路径、输出格式或日志格式，否则会影响实验结果的一致性。


4. Goal-Driven Execution：目标驱动执行

第四条原则是 Goal-Driven Execution，也就是围绕可验证目标执行。

它强调：AI 不应该只说“我改好了”，而应该说明：

- 这次修改的目标是什么；
- 修改后如何判断成功；
- 是否运行了测试；
- 如果没有测试，至少给出明确的检查方式；
- 如果是 bug 修复，最好先复现 bug，再修复 bug。

这条原则把 AI 编程从“生成代码”提升为“完成一个可验证的软件任务”。

例如：

用户说：“修复 CSV 处理脚本中的错误。”
更好的流程应该是：

1. 找到导致错误的输入样例；
2. 复现错误；
3. 修改最小代码；
4. 重新运行样例；
5. 确认输出符合预期。


四、项目结构与使用方式

这个项目原本是为 Claude Code 准备的。它的定位是一个单独的 CLAUDE.md 文件，用于改善 Claude Code 的行为。

如果在 Claude Code 中使用，可以直接把 CLAUDE.md 放到项目根目录，Claude Code 会根据这个文件中的规则工作。

但如果我们要在 Codex 中使用，就不能完全照搬 Claude Code 的插件方式。更合适的做法是把它转换成 Codex 能识别的 Skill 结构。

我们之前已经把它转换成了类似下面的结构：

karpathy-coding-discipline/
├── SKILL.md
└── Reference/
    └── original-guidelines.md

其中：

- SKILL.md 是 Codex 主要读取的技能说明文件；
- Reference/original-guidelines.md 保存原始规则内容，作为参考材料。

使用时可以在 Codex 中这样提示：

Use the karpathy-coding-discipline skill to fix this bug with minimal changes.

或者中文表达：

请使用 karpathy-coding-discipline skill，以最小改动修复这个问题。

这样 Codex 在执行任务时，就会优先遵循这套规则。


五、这个项目的实际价值

我认为这个项目的价值不在于它提供了复杂算法，而在于它提供了一套非常实用的 AI 编程行为约束范式。

它特别适合以下场景：


1. 科研代码修改

例如修改实验代码、评估脚本、数据处理脚本时，最重要的是稳定和可复现。
这个 skill 可以提醒 AI 不要随意修改数据路径、指标定义、输出格式。


2. Bug 修复

在 bug 修复场景中，它可以让 AI 先定位问题，再进行最小修改，而不是直接大范围重构。


3. 老项目维护

对于已有项目，如果 AI 随便重构，会带来很高风险。
这个 skill 强调 surgical changes，可以减少无关修改。


4. 代码审查友好

因为修改范围更小，diff 更干净，后续人工 review 会更容易。


六、与普通 Prompt 的区别

这个项目看起来像是一段 prompt，但它和临时 prompt 不完全一样。

普通 prompt 通常是一次性的，例如：

请帮我修复这个 bug。

而这个 skill 是一种可复用的行为规范。
它可以在多个任务、多个项目中反复使用，让 AI 始终保持相似的工作风格。

也就是说，它不是解决某一个具体编程问题，而是改善 AI 处理所有编程任务时的行为模式。


七、局限性

这个项目也有一些局限。

第一，它不是自动保证代码正确的工具。
它只是约束 AI 的行为，最终代码是否正确仍然需要测试和人工检查。

第二，它偏保守。
对于一些探索性任务，比如快速原型开发，它可能会让 AI 显得比较谨慎，不会主动做大规模设计。

第三，它本身不是 Codex 原生项目。
原始项目是为 Claude Code 准备的，如果要用于 Codex，需要转换成 SKILL.md 结构，或者改写成 AGENTS.md。

第四，它不能替代项目级规则。
例如科研项目中“不要修改指标定义”“不要改变数据格式”等规则，仍然需要我们自己补充。


八、总结

总体来看，andrej-karpathy-skills 是一个轻量但实用的 AI 编程辅助项目。

它的核心思想可以总结为四句话：

1. 写代码前先理解任务；
2. 优先使用简单方案；
3. 只修改必要代码；
4. 用测试或明确标准验证结果。

它的价值主要体现在：
让 AI 编程助手从“会写代码”变成“更像一个谨慎的软件工程协作者”。

对于我们的使用场景，尤其是科研代码、实验脚本、数据处理流程和项目维护，这个 skill 可以有效降低 AI 乱改代码、过度设计和缺少验证的问题。
