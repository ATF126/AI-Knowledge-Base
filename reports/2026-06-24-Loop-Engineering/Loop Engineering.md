# Loop Engineering 介绍

## 1. 概念理解

Loop Engineering，即循环工程，关注的是：怎么让 AI 在目标明确后，自己完成发现问题、拆任务、执行、检查、记录、继续下一步这一整套流程。

它不是简单地把提示词写得更长，也不是让 AI 重复跑很多遍，而是把一个任务设计成可以持续运行、可以自我检查、可以记录状态、也可以在必要时停下来的工作循环。

## 2. 从 Prompt 到 Loop：Agent 工程范式的演进

AI Agent 的工程方式正在从“人一步步指挥”走向“人设定目标，AI 自己跑循环”。

| 阶段 | 代表年份 | 核心问题 | 人的角色 | 主要痛点 |
| --- | --- | --- | --- | --- |
| Prompt Engineering | 2024 | 怎么写好一句话让 AI 听懂？ | 提示者 | 换任务就要重写提示词，复杂任务要反复对话 |
| Context Engineering | 2025 | 怎么把相关资料放进上下文？ | 资料组织者 | 上下文有限，放多了浪费 token，放少了 AI 不懂背景 |
| Harness Engineering | 2026 初 | 怎么给单个任务搭运行沙盒？ | 工具和权限设计者 | 每个任务都要单独搭环境，复用性有限 |
| Loop Engineering | 2026 年 6 月起 | 怎么让 AI 在多个工具、多个 Agent、多轮任务之间自动循环？ | 目标设定者和验收者 | 循环质量、成本、边界和验收成为重点 |

一句话总结这四年变化：

> 人写提示词 -> 人喂上下文 -> 人搭沙盒 -> 人定目标，AI 自己跑完剩下的循环。

## 3. Loop Engineering 到底在循环什么？

Loop Engineering 循环的不是让 AI 多跑几遍，而是一整套**决策 - 执行 - 验证 - 再决策**的工作流。

一个典型例子：

你说：“帮我分析本周 AI 领域热点，出一份周报。”

在 Harness 阶段，你可能要先搭一个热点监控 Agent、一个数据汇总 Agent、一个周报生成 Agent，再手动写它们之间的调用逻辑。

在 Loop 阶段，你只说目标，系统会自动做几件事：

1. 拆解任务：先监控热点，再收集数据，再分析，再写周报。
2. 分派角色：把不同子任务交给不同 Agent。
3. 调用工具：访问网页、数据库、文档、API 或本地文件。
4. 检查结果：发现缺信息、格式不对、事实不稳时自动返工。
5. 记录状态：记住哪些已经做完，哪些还没完成。
6. 回到人类：最终把结果交给人验收。

整个过程中，人只需要在开始说清目标，在结束判断结果是否合格。中间的调度、异常处理、重复执行和记录，由 loop 系统承担。

## 4. 支撑 Loop 跑起来的关键组件

一个完整 loop 至少要回答六个问题。

### 4.1 谁来唤醒它？

也就是 Automation，自动触发器。

loop 不能只靠人手动点一次。它要知道什么时候启动，比如：

- 每天固定时间扫描热点新闻。
- 每半小时检查一次新任务。
- 有用户反馈进来时自动分类并总结。

调度可以是定时，也可以由事件触发，还可以在目标达成后停止。

### 4.2 多个 Agent 怎么并行工作？

也就是 Worktree 或隔离工作区。

多个 Agent 同时工作时，很容易互相覆盖文件、改坏同一处代码，或者让结果混在一起。工程上需要给每个 Agent 一个独立空间，各自完成任务，再合并结果。

在代码场景里，这通常对应 git worktree、独立分支、独立执行环境。

### 4.3 AI 怎么知道项目规则？

也就是 Skills，技能或项目知识。

提示词是临时指令，Skill 更像长期规则。它可以记录：

- 项目怎么启动。
- 目录结构和命名规范。
- 代码风格。
- 常见坑。
- 测试和发布流程。

没有这层规则，loop 每次启动都像新同事第一天入职，要重新认识项目。

### 4.4 它能不能连接外部系统？

也就是 Connectors，连接器。

一个只会看本地文件的 Agent 能做的事情有限。真正有用的 loop 往往需要连接：

- issue 系统。
- 数据库。
- CI 和测试环境。
- PR 和代码仓库。
- 通知工具。
- 内部知识库。

这决定了 AI 是只能“提出建议”，还是能真正“推进工作”。

### 4.5 谁来检查结果？

也就是 Sub-agents，子 Agent 分工。

Loop Engineering 很强调“执行”和“验证”分离。写代码的 Agent 不应该只靠自己判断“我写得没问题”。更稳的做法是：

- 一个 Agent 负责实现。
- 一个 Agent 负责审查。
- 一个 Agent 负责测试或校对。
- 必要时换模型、换视角、换规则再查一遍。

这和团队协作里的 code review 很像：做事的人，最好不要自己给自己盖章。

### 4.6 它怎么记住进度？

也就是 State/Memory，状态与记忆。

长期 loop 离不开外部记忆。它至少要记录：

- 已经做过什么。
- 哪些结果已经验证。
- 哪些方案失败过。
- 哪些问题还需要人处理。
- 下一轮从哪里继续。

一句话总结：模型会忘，但仓库不会忘。对 loop 来说，Markdown 文件、看板、数据库、任务系统，都可以成为记忆载体。

## 5. 对用户意味着什么？

最直接的变化是：提示词的重要性会下降，但“目标定义”和“结果验收”的重要性会上升。

以前用户像输入端工程师，要努力把每句话说清楚。以后用户更像输出端质检员，要能判断：

- 我到底要什么？
- 合格结果长什么样？
- 哪些边界不能越过？
- 哪些错误必须返工？
- 什么情况下要停止，交给人处理？

所以人的价值不是消失，而是重新分配：

- 从反复动手，转向定义目标。
- 从一步步指挥，转向设计流程。
- 从盯着过程，转向验收质量。
- 从写提示词，转向设计 loop。

## 6. 为什么 2026 年才开始可行？

过去的自主 Agent 概念并不新，但早期模型能力撑不住，容易死循环、幻觉执行、乱调工具，或者跑着跑着回不来。

到 2026 年，有两个变化让 Loop Engineering 更接近落地：

1. 主流编程 Agent 能力大幅提升。Claude、Codex 等 Agent 在长链路任务中更能保持目标一致，知道什么时候进入下一步。
2. Harness 层基础设施成熟。工具调用、权限控制、MCP、沙盒、PR、CI、任务系统等基础设施已经能支撑单任务执行，Loop 层可以更专注于循环编排。

## 7. 成本和边界

Loop Engineering 不是万能方案。

### 7.1 成本可能爆炸

loop 会反复读上下文、反复尝试、反复验证，甚至拉多个 Agent 一起工作。Token、工具调用、计算资源和等待时间都可能成倍增长。

如果任务很简单，或者结果没有稳定反馈，硬上 loop 可能不划算。

### 7.2 Agent 会漂移

长循环里，AI 可能逐渐忘掉最初目标，或者把“不该做 X”的规则在第几十轮后丢掉。它还可能自我美化，给自己的输出打高分。

这要求我们设计停止条件、检查条件和外部审查机制。

### 7.3 会产生理解债务

loop 跑得越多，人越容易不看过程，只看一句“完成了”。但如果人不理解结果是怎么来的，代码、文档和决策会堆出新的债务。

一句话：AI 干得越多，人越要保留判断、验收和暂停权。

## 8. 哪些工作适合做成 loop？

适合 loop 的工作通常有三个特征：

1. 重复发生。
2. 流程相对稳定。
3. 结果有办法检查。

不太适合的场景：

- 一次性小任务。
- 标准不清的创意探索。
- 高风险、不可回滚的操作。
- 没有验收标准的模糊目标。

## 9. 一个最小可用 Loop 模板

可以把一个 loop 设计成下面这个结构：

```text
目标：要完成什么？
输入：需要哪些资料、数据、工具？
触发：什么时候启动？
拆解：分成哪些步骤或子任务？
执行：由哪个 Agent 或工具完成？
验证：用什么标准判断对错？
记忆：把进展、失败、结论记录到哪里？
停止：什么时候结束，什么时候交给人？
```

示例：每日 AI 前沿动态 loop

```text
目标：每天整理一份 AI 前沿动态摘要。
输入：新闻源、论文站、博客、社交媒体、产品更新页。
触发：每天固定时间启动。
拆解：采集信息 -> 去重筛选 -> 摘要整理 -> 标注不确定信息 -> 生成日报。
执行：采集 Agent、分析 Agent、写作 Agent。
验证：来源可追溯，事实有依据，重要信息不遗漏。
记忆：写入 Markdown 日志或任务系统。
停止：日报生成后停止；遇到重大不确定信息时交给人确认。
```

## 10. 分享时的核心结论

Loop Engineering 的重点不在“新名词”，而在协作方式变化：

- Prompt Engineering 解决“怎么说”。
- Context Engineering 解决“给它看什么”。
- Harness Engineering 解决“让它在哪里干活”。
- Loop Engineering 解决“让它如何持续、可靠、可检查地干活”。

未来真正稀缺的能力，可能不是写出最漂亮的提示词，而是设计一个稳定、可控、能交付结果的 AI 工作循环。
