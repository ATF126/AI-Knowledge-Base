# Harness Engineering 

## 一、主题概述

Harness Engineering 所讨论的核心问题，不是单纯如何使用模型，而是当 AI Agent 具备持续工作、调用工具、执行复杂任务的能力之后，如何通过系统设计，使其长期稳定、可控、可验证地运行。

这一主题所体现的基本判断可以概括为：

> **模型决定能力上限，系统决定落地质量。**

随着大模型能力不断提升，影响 Agent 实际可用性的关键因素，已经不再仅仅是模型本身的推理水平，而越来越取决于围绕模型构建的运行机制、约束体系、验证路径和治理结构。  
因此，Harness Engineering 可以被理解为 Agent 时代工程重心上移的体现，即从“优化单步输出”转向“治理整条执行链路”。

---

## 二、发展脉络

Harness Engineering 的形成并非孤立现象，而是 AI 工程在近几年持续演化后的结果。其发展过程大致可以概括为以下几个阶段。

## 1. 生成阶段 2022.11 — 2023 
这一阶段的重点在于模型是否能够完成高质量生成。  
工程实践主要围绕 **Prompt Engineering** 展开，核心关注点是：

- 提示词设计
- few-shot 示例构造

这一时期的模型已经具备较强生成能力，但仍停留在“能回答、不能执行”的层面。  
工程问题主要集中在单次调用质量，而非长期任务控制。

产物：Prompt Engineering

## 2. 连接阶段 2023 — 2024 
随着工具调用、外部 API 接入和插件体系发展，模型逐渐从“能说”发展到“能连接”。

这一阶段带来的变化在于：
- 模型开始具备调用外部工具的能力；
- 可访问的数据与服务范围大幅扩展；
- 应用场景从单纯问答转向任务执行。

> **问题：连接不等于编排，编排不等于治理。**

工具虽然接上了，但围绕工具的选择、调用顺序、错误处理、结果验证和回退机制并不成熟，因此系统稳定性仍然较弱。

产物：LangChain

## 3. 推理阶段 2024 
推理模型能力增强之后，单步任务质量显著提高。  
但与此同时，一个更加清晰的事实逐渐显现：

> **问题：单步推理更强，并不意味着长链路任务会自动变得可靠。**

在长任务、多步骤、跨上下文的场景中，系统依然会出现：
- 目标漂移
- 多轮执行不一致
- 长任务中断后无法续航

因此，问题开始从“如何让模型更聪明”转向“如何让系统更稳定”。

产物：Context Engineering

## 4. 行动阶段 2025
随着各类 Agent 产品快速出现，模型开始真正参与多步骤任务执行。  
这一阶段的突出特征是：

- Agent 可以调用多种工具；
- 可以在较长时间内持续推进任务；
- 可以承担部分复杂工作流。

但工程上的主要发现并不是 Agent 能做什么，而是 Agent 在复杂任务中如何失控。  
典型问题包括：

- 上下文耗尽
- 中途偏离目标
- 提前宣布完成
- 工具调用冗余或错误
- 持续生成导致代码库和文档质量下降

> **问题：Agent的问题已不再是能力缺失，而是治理缺失。**

## 5. 治理阶段 2026 
进入这一阶段后，工程关注点明显转向 Agent 的可控性、稳定性和长期治理能力。  
Harness Engineering 正是在这一背景下逐渐成形的。

这一阶段的核心转向可以表述为：

> **研究重点：不再只是让 Agent 更能干，而是让 Agent 在长时间、复杂环境、多工具条件下仍然不失控。**

产物：Harness Engineering

---

Harness Engineering 的出现，也体现出工程抽象层级的明显上移。  
这一演化链条可以概括为：

- Prompt Engineering：关注“单次调用如何提问”
- Context Engineering：关注“每一步应该给模型什么信息”
- Harness Engineering：关注“整条执行链路如何运转、如何验证、如何续航、如何治理”

因此，Harness Engineering 的出现，标志着 AI 工程重点从模型调用技巧进一步上移到了系统设计层。

## 三、Agent 的挑战

Harness Engineering 的必要性，主要来自 Agent 在开放环境中运行时所面临的几个根本性挑战。

## 1. 状态持久性
本质：Agent 需要跨时间、跨 session 记住做过什么。 
为什么模型解决不了：模型本身是无状态的，context window 也有上限，无法天然承担长期连续状态。

## 2. 目标一致性
本质：长任务中，agent 容易漂移、自嗨，甚至提前宣布完成。
为什么模型解决不了：模型缺少外部锚点，无法稳定校准“什么才算真正完成”。

## 3. 行动可验证性
本质：每一步都是概率性的，需要区分“做了”和“做对了”。
为什么模型解决不了：模型在评价自己结果时，天然存在自我表扬和误判倾向。

## 4. 熵增抑制
本质：持续产出会不断累积冗余、漂移和不一致。
为什么模型解决不了：模型会复制已有模式，哪怕这些模式本身就是坏的或低质量的。

## 5. 人机边界
本质：何时自主、何时交给人，需要明确且工程化地定义。
为什么模型解决不了：模型无法稳定判断什么时候该停下来交还人类。

## 四、Harness 的基本定义

Harness 可定义为：

> **支撑模型作为 Agent 持续行动的外循环系统。**

这一外循环系统并不等同于提示词，也不等同于某一个框架，而是指围绕模型运行所建立的一整套任务推进与治理机制。  
其典型内容包括：

- 计划与任务分解
- 持久状态管理
- 工具编排与调用中介
- 反馈回路
- 回退机制
- 人机交接点
- 审计与记录机制

需要特别区分的是，Harness 所强调的是 **运行时外循环**，而不是单纯的评测流水线。  
因此，Agent 的效果不能仅归因于模型本身，而应理解为 **model + harness** 的组合结果。

[*Agent = LLM + Harness*](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents?ref=monabellrose)

## 五、Harness 的工程构件

Harness Engineering 的方法论价值，主要体现在其对多个关键工程构件的系统组织上。

## 1. Durable State Surfaces
> **问题： 长 Agent 的核心痛点，就像一个项目组里的工程师每次换班都完全失忆。Context window 有限，复杂项目无法在单个窗口内完成，Agent 需要一种方式桥接 session 之间的鸿沟。**

解决办法：不是简单保留聊天记录，而是将状态沉淀为结构化、可续航的外部工件
- init.sh：怎么启动环境
- progress.txt：当前做到哪了
- 初始 git commit：最开始的基线状态
- feature list：有哪些功能要做、哪些还没完成

> **原理：Durable State Surfaces 就是给 Agent 建“可交接的工作面”，让它不用靠记忆接任务，而是靠结构化工件续上任务。**

## 2. Decomposition & Plans
> **问题： 把一个很大的目标直接扔给 Agent ，但任务太大、太散、太长，超出了它一次稳定处理的范围。**

解决办法：角色系统
- Planner 不直接执行任务，而是把一两句高层描述扩展成完整 product spec 和分步 feature list
- Generator 负责逐 feature 落地，每完成一个就 commit
- Evaluator 负责独立评估 generator 的产出，标记 pass/fail，给出具体改进建议

> **原理：把大目标拆成可执行、可验证、可交接的小块，并把计划变成可以保存、继承和引用的系统工件。**

## 3. Feedback Loops
> **问题： Agent 写了代码，怎么知道写得对不对？**

解决办法：
- 在 agent 行动之前约束
- 在 agent 行动之后进行自纠错
[*Agent 不应该孤零零地生成一份结果，而应该被放进一个持续校验和纠偏的流水线里*](https://martinfowler.com/articles/harness-engineering.html)

> **原理：Feedback Loops 的本质，是让 Agent 不靠自我感觉判断质量，而是被放进一个有前馈约束、后馈检查和持续纠错的外部控制系统里。**

## 4. Legibility
> **问题： Agent 能写代码了，但它能“看到”自己写的代码跑起来是什么样吗？**

解决办法：
- 通过浏览器实例，让 Agent 看到 UI
- 把 logs、metrics、traces 全部给 Agent 
- 把知识库里最重要的知识记录下来
- 把给 Agent 看的说明书、操作手册、任务计划，也都放进项目仓库里

> **原理：Legibility 的本质，是把原本只对人类可见的知识、规则、信号和历史，转化成 Agent 在运行时真正可访问、可理解、可利用的感知面。**

## 5. Tool Mediation
> **问题： Agent 可能访问上百个工具。但直接把所有工具定义都塞进上下文，会导致 Token 暴增、延迟上升。**

解决办法：
- 模型先写一段代码
- 这段代码在执行环境里运行
- 代码按需发现和调用工具
- 数据过滤、中间处理、结果整理都在外部完成
- 最后只把需要的结果回给模型

> **原理：把工具使用从模型的内循环，挪到更高效的外部执行回路里。**

## 6. Entropy Control
> **问题： Agent 不只是会帮你干活，它也会持续制造系统垃圾，不断放大系统噪声。**

解决办法：
- Documentation consistency agents：定期验证文档与代码是否一致
- Refactor agents：按计划清理冗余信息
- Architectural enforcement：通过特定的规则维护模块边界

> **原理：Harness 不只负责让 agent 跑起来，还负责持续抑制 agent 放大的系统噪声**

## 六、这一方向所反映的更深层变化

这一主题的重要性，不只在工程技巧层面，更在于它揭示了一个更深层的范式变化：  
**系统正在从指令驱动走向意图驱动。**

在传统指令驱动模式下，系统按人给出的明确步骤执行。  
而在 Agent 模式下，人类更多只表达目标，执行路径由系统自行规划。

这种变化带来了新的控制问题：

- 错误不再只是“没按指令执行”
- 错误还可能是“误解意图”或“虽然理解了意图，但选择了糟糕的路径”

因此，Harness Engineering 的本质，可以理解为对意图驱动系统所必需的新型控制层的工程回应。

---

## 七、仍需继续关注的问题

尽管 Harness Engineering 已经提供了较强的解释框架，但仍存在若干开放问题。

## 1. Harness 本身如何验证
如果外循环中的验证器本身也具有不确定性，那么系统可能会进入“用不可靠系统验证不可靠系统”的循环。  
因此，Harness 的可测试性和可靠性本身也是关键问题。

## 2. 多 Agent 协作的复杂性
单 Agent 场景相对容易控制，而多 Agent 并行、协作、共享状态时，会出现更复杂的涌现行为。  
这一问题在未来可能逐渐接近分布式系统治理问题。

## 3. 成本与收益如何平衡
Planner、evaluator、sensor、重构与审计机制都会增加成本和时延。  
因此，如何根据任务复杂度动态确定 Harness 深度，是一个重要工程课题。

## 4. 安全边界如何定义
随着 Agent 拥有更多工具权限和持续自治能力，安全问题将不再只是回答错误，而可能变成越权执行、跨工具数据流失和错误行为放大。  
因此，Harness 也将逐渐成为新的安全边界。

---

## 八、总结

Harness Engineering 所体现的，并不是某一个局部技巧，而是一种新的工程重心。  
其核心意义在于：当 AI Agent 从短时、单轮、低风险的调用工具，发展到长时、多步骤、跨工具、可持续执行的自治系统时，仅靠模型能力已经不足以支撑系统稳定运行，必须通过外循环设计建立起状态、计划、验证、治理与审计能力。

因此，这一方向可以概括为以下判断：

> **未来真正具有竞争力的 AI 系统，不只是更聪明，而是更可治。**

从这个角度看，Harness Engineering 的价值并不局限于当前 Agent 实践，而是在更深层上推动 AI 工程从“围绕模型优化”走向“围绕智能系统治理”。
